    **Session persistence (src/modules/timer/session.ts):**
    Handles DB operations for timer sessions. Import ExtendedPrismaClient, awardXP from xp/engine, XP_AWARDS from xp/constants.

    Export `persistTimerSession(db, timer: ActiveTimer, status: 'COMPLETED' | 'CANCELLED')`:
    1. Find active season: `db.season.findFirst({ where: { active: true } })`
    2. Calculate xpToAward:
       - If status is CANCELLED or totalWorkedMs < minSessionMinutes * 60_000: xpToAward = 0
       - Otherwise: rawXP = Math.floor(totalWorkedMs / 300_000) * xpPer5Minutes (1 XP per 5 min)
       - Get today's timer XP via getTodayTimerXP()
       - remaining = max(0, dailyCap - todayUsed)
       - xpToAward = min(rawXP, remaining)
    3. Create TimerSession record: `db.timerSession.create({ data: { ... all fields from ActiveTimer mapped to Prisma fields, status, xpAwarded, endedAt: new Date(), seasonId } })`
    4. If xpToAward > 0: `awardXP(db, memberId, xpToAward, 'TIMER_SESSION', 'Timer session: X min')`
    5. Return `{ xpAwarded, durationMinutes: Math.floor(totalWorkedMs / 60_000), leveledUp, newRank, oldRank }`

    Export `getTodayTimerXP(db, memberId)`:
    Follow the exact `getTodayVoiceXP` pattern from voice-tracker/tracker.ts:
    - todayStart = new Date() with setUTCHours(0,0,0,0)
    - Aggregate XPTransaction where source='TIMER_SESSION' and createdAt >= todayStart
    - Return _sum.amount ?? 0

    Export `createActiveTimerRecord(db, timer: ActiveTimer)`:
    Creates an ACTIVE TimerSession in DB for restart recovery. Called on timer start.
    Maps ActiveTimer fields to Prisma model. Sets status='ACTIVE'.

    Export `updateTimerRecord(db, memberId, updates)`:
    Updates the ACTIVE TimerSession for this member. Called on state transitions for restart recovery.
    Uses `db.timerSession.updateMany({ where: { memberId, status: 'ACTIVE' }, data: { ...updates, lastStateChangeAt: new Date() } })`.

      - Option 'break': Integer, description 'Break duration in minutes (default 5, pomodoro only)', min 1, max 60, not required
      - Option 'focus': String, description 'What are you working on?', not required, max 100 chars
      - Option 'goal': String, description 'Link to an active goal', not required, autocomplete true
    - Subcommand 'pause': no options
    - Subcommand 'resume': no options
    - Subcommand 'stop': no options
    - Subcommand 'status': no options

    Export `handleTimerCommand(interaction, ctx)`:
    Defer ephemeral reply immediately. Resolve memberId from discordId via db.discordAccount.findUnique.
    Route by subcommand name:

    **start:**
    1. Check if member already has an active timer -- if so, reply "You already have a timer running. Stop it first with /timer stop."
    2. Read options: mode (default 'pomodoro'), work (default 25), break (default 5), focus, goal
    3. If goal option provided: resolve via db.goal.findFirst({ where: { id: goalId, memberId, status: 'ACTIVE' } }). If not found, reply error. Set focus to goal title, goalId to goal.id.
    4. For proportional mode: breakDuration is calculated dynamically (ignore the break option). Set breakDuration to 0 as placeholder.
    5. Call createTimer(memberId, { mode, workDuration, breakDuration, breakRatio: TIMER_DEFAULTS.defaultBreakRatio, focus, goalId })
    6. Send DM to user: create DM channel, send message with buildTimerEmbed(timer) + buildWorkButtons(). Store dmMessageId and dmChannelId on the timer.
    7. Create ACTIVE DB record via createActiveTimerRecord(db, timer)
    8. Schedule work->break transition via scheduleTransition(memberId, workDuration * 60_000, async callback). The callback: call transitionToBreak, edit DM message with updated embed + buildBreakButtons(), send break start message content, schedule next transition (break->work for pomodoro, or idle timeout for proportional).
    9. Edit ephemeral reply: "Timer started! Check your DMs."
    10. Emit timerStarted event.

    **pause / resume / stop:**
    Call the corresponding engine function. Edit the DM message with updated embed and appropriate buttons. For stop: call persistTimerSession, show completed embed with XP, emit timerCompleted or timerCancelled. Update DB record via updateTimerRecord on pause/resume.

    **status:**
    Call getActiveTimer. If none, reply "No active timer." If active, reply with buildTimerEmbed showing current state including calculated remaining time.

    **Global button routing (src/index.ts):**
    Add `isButton()` check to the existing `interactionCreate` handler, BEFORE the `isChatInputCommand()` check, AFTER the `isAutocomplete()` check. Exactly as specified in 09-RESEARCH.md Pattern 3:
    ```
    if (interaction.isButton()) {
      events.emit('buttonInteraction', interaction);
      return;
    }
    ```
    This emits buttonInteraction on the event bus so any module can listen for button presses. The timer module will filter by customId prefix.

    **Timer module (src/modules/timer/index.ts):**
    Follow the voice-tracker/index.ts pattern exactly. Export a default Module object with name 'timer' and register(ctx) function.

    In register(ctx):
    1. Cast db as ExtendedPrismaClient.

    2. **Import and implement withMemberLock** (copy the pattern from ai-assistant/chat.ts):
       - `const processingLocks = new Map<string, Promise<void>>()`
       - `async function withMemberLock<T>(memberId: string, fn: () => Promise<T>): Promise<T>` -- same implementation as chat.ts. This prevents race conditions on rapid button presses (Pitfall 2 from research).

    3. **Register commands:**
       `ctx.commands.register('timer', buildTimerCommand(), handleTimerCommand);`

    4. **Register autocomplete handler:**
       ```
       ctx.events.on('autocomplete', async (...args: unknown[]) => {
         const interaction = args[0] as AutocompleteInteraction;
         if (interaction.commandName !== 'timer') return;
         await handleTimerAutocomplete(interaction, db);
       });
       ```

    5. **Register button interaction handler:**
       ```
       ctx.events.on('buttonInteraction', async (...args: unknown[]) => {
         const interaction = args[0] as ButtonInteraction;
         if (!interaction.customId.startsWith('timer:')) return;
         await interaction.deferUpdate(); // Respond within 3 seconds (Pitfall 1)
         await handleTimerButton(interaction, db, ctx);
       });
       ```

    6. **Implement handleTimerButton(interaction, db, ctx):**
       Extract memberId from interaction.user.id -> db.discordAccount lookup.
       Wrap all mutations in withMemberLock(memberId, async () => { ... }).

       Route by interaction.customId:
       - `timer:pause`: call pauseTimer, update DB record, edit message with paused embed + buildPausedButtons()
       - `timer:resume`: call resumeTimer, update DB record, edit message with working/break embed + appropriate buttons, reschedule transition timeout for remaining time
       - `timer:stop`: call stopTimer, persist session via persistTimerSession, edit message with completed embed (no buttons -- remove components), emit timerCompleted/timerCancelled
       - `timer:skip_break`: call transitionToWork, update DB record, edit message with working embed + buildWorkButtons(), schedule next work->break transition

       On any error: try to edit the message with an error note, log the error.

    7. **Implement transition orchestration helper:**
       `async function orchestrateTransition(memberId: string, db, ctx)`:
       Called after engine state changes that need DM updates and timeout scheduling.
       - Get the ActiveTimer
       - Get the DM channel and message from stored IDs
       - Edit the message with updated embed and buttons
       - If state is 'working': schedule work->break transition for workDuration
       - If state is 'on_break' (pomodoro): schedule break->work transition for breakDuration
       - If state is 'on_break' (proportional): schedule idle timeout (no auto-resume for proportional)

       For work->break transition callback:
       - Call transitionToBreak on engine
       - Edit DM with break embed + break buttons
       - Send new message content with buildBreakStartMessage (the encouraging text)
       - Update DB record
       - If pomodoro: schedule break->work transition
       - If proportional: schedule idle nudge then auto-end

       For break->work transition callback (pomodoro only):
       - Call transitionToWork on engine
       - Edit DM with work embed + work buttons
       - Send message content with buildWorkResumeMessage ("Back to it!")
       - Update DB record
       - Schedule next work->break transition

       For idle timeout callback (no-return handling per user decision):
       - After break ends + gentleNudgeDelayMinutes: send a single gentle DM ("Your break ended a few minutes ago. Ready to continue or done for now?")
       - After break ends + idleTimeoutMinutes: auto-end the session. Call stopTimer, persistTimerSession as COMPLETED (they worked, just didn't return from break), edit DM with completed embed.
       - Priority is NOT distracting workflow -- one nudge maximum, then auto-end.

    8. **Restart recovery (client.once 'ready'):**
       Query db.timerSession.findMany({ where: { status: 'ACTIVE' } }) with member relation.
       For each active session:
       - Calculate elapsed time since lastStateChangeAt
       - Determine current state (if was working and elapsed > workDuration, they should be on break, etc.)
       - Restore in-memory ActiveTimer via restoreTimer()
       - Send a NEW DM message (don't try to edit old one -- Pitfall 5) with current state embed and buttons
       - Store new dmMessageId/dmChannelId
       - Schedule appropriate transition timeout for remaining time
       - Log count of reconstructed timers

    9. Log module registration: `ctx.logger.info('[timer] Module registered');`
