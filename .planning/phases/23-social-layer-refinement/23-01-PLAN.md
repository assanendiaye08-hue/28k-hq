    - Keep `graceDaysPerWeek: 2` as-is (weekly grace budget unchanged)
    - Keep `recoverBonus: 15` as-is

    **streak.ts two-day rule changes:**
    - In `updateStreak()`, change the reset threshold from `daysSinceLastCheckIn >= 7` to `daysSinceLastCheckIn > STREAK_CONFIG.missedDaysToBreak` (i.e., > 2 means 3+ days missed = break)
    - CRITICAL: Never set `currentStreak = 0`. When streak breaks, set `currentStreak = 1` (the current check-in starts a new streak). This is already the case for the 7+ day branch -- verify and maintain.
    - For 2 days missed (daysSinceLastCheckIn === 2): this is within grace -- streak maintained via existing grace day logic (no change needed here)
    - For 3+ days missed: streak breaks. Set `currentStreak = 1`, `isComeback = true`, award `recoverBonus`
    - Remove the 2-6 day branch that does multiplier decay via reducing effective streak count -- replaced by cleaner two-day rule
    - New logic flow:
      ```
      daysSinceLastCheckIn === 0 -> same day, no change
      daysSinceLastCheckIn === 1 -> consecutive, increment
      daysSinceLastCheckIn === 2 -> 1 missed day, streak maintained (within two-day rule)
      daysSinceLastCheckIn >= 3 -> 2+ missed days, streak breaks to 1, comeback bonus
      ```
    - Grace days per week logic (calculateGraceDays) can be REMOVED since the two-day rule supersedes it. The two-day rule is simpler and more intuitive: miss 1 day = fine, miss 2+ = break. No need for weekly grace budgets.

    **Add `calculateConsistencyRate()` to streak.ts:**
    - New exported function: `calculateConsistencyRate(db, memberId, windowDays = 30): Promise<number>`
    - Query CheckIn records for the member within the last `windowDays` days
    - Count distinct days with at least one check-in
    - Return `(daysWithCheckIns / windowDays) * 100` rounded to nearest integer
    - Example: 26 check-in days out of 30 = 87%

    **Update StreakUpdateResult interface:**
    - Add `consistencyRate: number` field
    - Populate it by calling `calculateConsistencyRate()` at end of `updateStreak()`

    **Update check-in response (commands.ts):**
    - After streak update, include consistency rate in the embed response
    - Add a field like: "Consistency: 87% (30d)" alongside the existing streak display
    - If consistency >= 90%: add a small note like "Excellent consistency"
    - If consistency >= 70%: show the rate normally
    - If consistency < 70%: show but with encouraging note like "Building momentum"

    **Update leaderboard streak renderer (renderer.ts):**
    - In `buildStreakLeaderboardEmbed()`, show consistency rate alongside streak count
    - Format: `{streak}d streak | {rate}% (30d)` for each entry
    - This requires the renderer to call `calculateConsistencyRate` for each top member
    - Update the function signature to accept `db: ExtendedPrismaClient` parameter
    - Update all callers in channel-sync.ts to pass `db`

    **Update leaderboard calculator (calculator.ts):**
    - The streak leaderboard sorts by `currentStreak` -- keep this as primary sort
    - No structural change needed since consistency rate is displayed in renderer, not used for ranking
