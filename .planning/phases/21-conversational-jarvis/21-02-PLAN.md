     c. `set_reminder`: description "Set a reminder when the user asks to be reminded about something at a specific time". Parameters: `content` (string, required), `time` (string, required -- natural language), `recurring` (string, optional -- pattern like "every Monday").
     d. `track_commitment`: description "Track a commitment when the user says they will do something by a specific time -- statements like 'I will', 'I'll have', 'I'm going to' with a deadline". Parameters: `what` (string, required), `by_when` (string, required).
     e. `start_brainstorm`: description "Start a structured brainstorming session when the user explicitly asks to brainstorm, explore ideas, or think through a problem creatively". Parameters: `topic` (string, required).
   - Each tool description should be specific enough to avoid false positives (see research pitfall 1). Include negative examples in descriptions where helpful.

2. **intent-executor.ts** -- Confirmation flow and action execution:
   - Export `PendingAction` interface: `{ tool: string; params: Record<string, unknown>; confirmMessage: string; createdAt: number }`
   - Export `pendingActions` Map<string, PendingAction> keyed by memberId
   - Export `PENDING_ACTION_TTL = 5 * 60 * 1000` (5 minutes)
   - Export `isConfirmation(text: string): boolean` -- matches: yes, yeah, yep, sure, ok, do it, go ahead, confirm, y (case insensitive, word boundary at start)
   - Export `isDenial(text: string): boolean` -- matches: no, nah, nope, cancel, never mind, n (case insensitive, word boundary at start)
   - Export `buildConfirmation(toolName: string, params: Record<string, unknown>): string` -- returns conversational confirmation for each tool type. Make confirmations natural, NOT clinical. Examples:
     - log_checkin: `Logging check-in: "${activity}". Sound right?`
     - create_goal: `Creating goal: "${title}" due ${deadline}. Confirm?`
     - track_commitment: `Tracking: you'll "${what}" by ${by_when}. Got it?`
     - set_reminder: `Reminder: "${content}" at ${time}. Set it?`
     - start_brainstorm: (no confirmation needed -- brainstorm starts immediately)
   - Export `executePendingAction(db, memberId, action, message): Promise<void>` -- executes the confirmed action:
     - **log_checkin**: Reuse existing patterns from checkin/commands.ts:
       1. Call `extractCategories(db, memberId, activity)` from `../checkin/ai-categories.js`
       2. Get member schedule for timezone
       3. Count today's check-ins (`db.checkIn.count` with `createdAt >= todayStart`)
       4. Create CheckIn record: `db.checkIn.create({ data: { memberId, content: activity, effortRating: effort ?? null, categories, dayIndex: todayCheckIns + 1 } })`
       5. Call `updateStreak(db, memberId, timezone)` from `../checkin/streak.js`
       6. Call `calculateCheckinXP(dayIndex, streakResult.multiplier)` and `awardXP(db, memberId, xp, 'CHECKIN', description)` from `@28k/shared`
       7. Reply with: `Logged: "${activity}" [${categories}]. +${xp} XP. Streak: ${streak} days.`
     - **create_goal**: Parse deadline using chrono-node (`import * as chrono from 'chrono-node'`). Create goal with `db.goal.create`. Reply with: `Goal set: "${title}" -- due ${formattedDeadline}.`
     - **set_reminder**: Use `parseReminder` from `../reminders/parser.js` to parse time. Create reminder with `db.reminder.create`. Schedule using `scheduleOneShot`/`scheduleRecurring` from `../reminders/scheduler.js`. Reply with formatted time.
     - **track_commitment**: Parse deadline with chrono-node. Create Commitment record: `db.commitment.create({ data: { memberId, title: what, deadline: parsedDeadline } })`. Reply with: `Tracked: "${what}" by ${formattedDeadline}.`
     - **start_brainstorm**: This case should NOT go through confirmation flow. If it appears here, just reply "Starting brainstorm..." (Plan 03 will handle the actual brainstorm flow).
   - Add cleanup: before checking pendingActions, check `createdAt` against `PENDING_ACTION_TTL` and delete expired entries.
   - Import the Discord `Message` type and `Client` type for the message parameter.
   - The executor needs access to the Discord `client` for reminder scheduling -- accept it as a parameter or access it through the message object.
