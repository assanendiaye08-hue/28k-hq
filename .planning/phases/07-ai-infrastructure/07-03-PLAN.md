  <action>
Evolve the existing `assembleContext()` function in memory.ts to use tiered context assembly. The existing function already loads member data, conversation summary, and recent messages. Enhance it with hot/warm/cold tiers and protected data.

**1. Define tier boundaries (as constants at top of file):**
- `HOT_WINDOW_DAYS = 7` -- last 7 days: data included verbatim
- `WARM_WINDOW_DAYS = 30` -- days 8-30: weekly summaries
- `COLD_THRESHOLD_DAYS = 30` -- 30+ days: monthly compressed summary
- These boundaries are conservative per user decision (Grok 2M context gives us headroom)

**2. Define protected data categories (never compressed):**
Add a comment block listing what is always included verbatim regardless of tier:
- Member's display name, work style, current focus, timezone, accountability level
- All active goals at any level
- Inspirations (when added in Phase 8 -- future-proof the list)
- Personal details from profile (interests, learning areas)
- Reflection breakthroughs (when added in Phase 12 -- future-proof)

These are already loaded by `assembleContext()` via the member include. The key change is ensuring they are NEVER part of the token-trimming loop.

**3. Evolve `assembleContext()` return type:**
Update the `AssembledContext` interface:
```typescript
export interface AssembledContext {
  summary: string | null;           // Cold tier: monthly compressed
  weeklySummaries: string[];        // Warm tier: weekly pattern summaries
  recentMessages: Array<{ role: string; content: string }>;  // Hot tier
  memberContext: string;            // Protected: always verbatim
}
```

**4. Enhance `assembleContext()` logic:**
The function already loads member data and recent messages. Enhance:

a) **Hot tier (last 7 days):** Load conversation messages from the last 7 days verbatim. Use `createdAt: { gte: sevenDaysAgo }` where `sevenDaysAgo = subDays(new Date(), HOT_WINDOW_DAYS)`. Import `subDays` from `date-fns`.

b) **Warm tier (days 8-30):** Load conversation messages from days 8-30 and group by week. For each week, if there are messages, create a short summary string like "Week of {date}: {message count} exchanges about {topic hints from content}". Use a simple heuristic -- take the first 200 chars of the concatenated messages as a preview. Do NOT make an AI call for warm summaries (that would be expensive and recursive). Just use a basic text truncation approach.

c) **Cold tier (30+ days):** Use the existing `ConversationSummary` record. This is the rolling summary already maintained by `compressSummary()`. No change needed for cold tier -- it already works.

d) **Protected data:** The `buildMemberContext()` function already assembles display name, goals, profile data, check-ins, voice sessions. Ensure this output is NOT included in the token-trimming loop. Move the token trimming to only affect hot-tier messages (same behavior as current, just explicit about what's trimmable).

e) **Token budget trimming:** Keep the existing logic that trims oldest messages when over budget, but:
  - Protected data (memberContext) is never trimmed
  - Weekly summaries are trimmed before hot messages (they're lower priority)
  - Hot messages are trimmed last (most recent = most valuable)
  - If still over budget after trimming warm + hot, trigger `compressSummary()`

**5. Evolve `compressSummary()` to use tiered approach:**
The existing function compresses all older messages into one rolling summary. Update it:
- When triggered, it should compress messages older than `WARM_WINDOW_DAYS` into the monthly summary (cold tier)
- Messages in the warm window (8-30 days) are NOT compressed by this function -- they stay as messages for the warm tier assembly
- Use `callAI()` from shared/ai-client instead of direct OpenRouter call (already done in Plan 02, just verify)

**6. Update callers of `assembleContext()`:**
Check which files call `assembleContext()` -- primarily `chat.ts` and `briefs.ts`. Update them to handle the new `weeklySummaries` field in the returned context. When building the messages array for the AI call:
- Include `memberContext` as part of the system prompt (already done)
- Include `summary` (cold) if present, prefixed with "Historical context: "
- Include `weeklySummaries` joined, prefixed with "Recent weeks: "
- Include `recentMessages` as the conversation history (already done)

**Transparency rule per user decision:** Never mention to the member that their context is compressed or tiered. Jarvis uses whatever context is available naturally.
